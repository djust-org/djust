# ADR-022: v1.1 Code-Quality Headline — Single-Path Convergence (Quality-First → ViewRuntime)

**Status**: Proposed — chosen 2026-06-23 via `/pipeline-strategy --deep` (see [`docs/strategy-sessions/2026-06-23-v1.1-code-quality.md`](../strategy-sessions/2026-06-23-v1.1-code-quality.md))

**Supersedes (direction only)**: the v1.1 headline choice in [`docs/strategy-sessions/2026-05-19-v1.1-readiness.md`](../strategy-sessions/2026-05-19-v1.1-readiness.md) (Path E — "defer the headline pending launch-soak data"). The launch soak (v1.0.0 → v1.0.8) is now complete; this ADR commits the headline that Path E deliberately left open. The LiveView Native track (ADR-019) and `dj-island` (ADR-020) / auto-navigation (ADR-021) remain valid in parallel — see Consequences.

## Context

The v1.0.x arc (v1.0.0 GA → v1.0.8) shipped 8 patch milestones plus the v1.0.7 security audit + v1.0.8 prevention program. Across that arc, **one failure class dominated every retro: `#1646` parallel-path-drift** — a control or behavior implemented per-transport (WebSocket / SSE / `ViewRuntime` / HTTP-API / HTTP-GET) that drifts when one path is changed and a parallel path is not. It appears **21× in `RETRO.md` and 12× in `CLAUDE.md`**, recurred **3× within the single v1.0.8-2 drain** and **4×** the cycle before, and was the root shape of the entire security audit (the systemic finding "SSE consistently lacks controls the WS transport has").

The v1.0.8-1 prevention program converged the *auth sequence* into one `run_pre_mount_auth` helper and shipped anti-drift nets (`#1859` transport-parity suite + mount-orchestration AST pins), but **deliberately deferred the full dispatch convergence** — migrating WS `handle_mount` + `_handle_event_inner` (~1000 lines, ~16 WS-only behaviors) and the SSE mount/event paths onto a single `ViewRuntime.dispatch` spine — soak-gating it under ADR/`#1122` (split-foundation) because it is the single highest-blast-radius change in the codebase. That deferral (private `#173` WU3/WU4) is now the residual root of the `#1646` class: as long as N dispatch paths exist, drift is *possible*; only collapsing them to one makes it *impossible*.

The user, at the v1.1 milestone boundary, steered v1.1 toward **code quality / DRY / single-path / refactor**. This ADR records the resulting directional decision.

## Decision

**v1.1's headline is a code-quality / single-path convergence arc, executed quality-first (strategy Path B):**

1. **Quality groundwork first (low-risk, continuous, builds the safety net).** Extend the anti-drift parity nets to every control; pay down concrete DRY/footgun debt (`#1360` dedupe `_parseTimeMs`/`_computeTransitionTiming`, `#1279` single `isSignificantChild`, `#1368` log-injection asymmetry in `rust_bridge`, `#1648` sync-async `check_handler_permission`, `#1356` state-backend shared-ref); canonicalize the opt-in framework patterns (`#1307`) + finish the bidirectional-binding inventory (`#1308`); de-flake the suite + promote the browser-smoke to a hard gate (`#1875`, `#1869`, `#1152`, `#1153`).
2. **Then the ViewRuntime convergence headline.** Once drift is *mechanically detectable* (the nets from step 1), migrate the WS + SSE dispatch onto the single `ViewRuntime` spine in split-foundation iterations (mount first, then event), retiring the `#1646` class structurally. Module-organization refactors (splitting `websocket.py` ~5000 lines / `live_tags.py`) ride alongside this step, not before it (reorganizing before converging would move the same code twice).

**Why quality-first and not convergence-first:** the convergence is the right destination, but landing a ~1000-line dispatch merge with ~16 WS-only behaviors *without* the parity nets in place is the exact "giant un-netted PR" risk that `#1122` (split-foundation) and `#1859` (anti-drift nets) exist to prevent. The v1.0.8-1 convergence was safe *because* the nets came first; v1.1 repeats that discipline at larger scope.

## Consequences

- **Positive**: retires the #1 recurring failure class of the v1.0.x arc at its root; each early iteration delivers a standalone quality win (no "big bang" dependency); the convergence lands on a net, not a hope; one dispatch spine means a future control is added once and lives on every transport.
- **Cost / risk**: the convergence step remains the highest-blast-radius change in the codebase — mitigated by (a) doing it last, after the nets, and (b) split-foundation iterations with per-iteration soak per `#1122`. The ~16 WS-only behaviors (binary upload, presence, cursor, time-travel, sticky-child preservation, signed-snapshot restore, actor channel-layer) must each be explicitly mapped onto runtime hooks — this is the work the v1.0.8-1 retro flagged as "not a thin shim."
- **Relationship to parallel ADRs**: ADR-019 (LiveView Native) Iter-I is *renderer abstraction + `ViewRuntime` plumbing* — it needs the same single `ViewRuntime` spine this convergence builds, so the two are **aligned, not competing**; sequencing the convergence first de-risks LVN-I. ADR-020 (`dj-island`) and ADR-021 (auto-navigation) are independent feature tracks and are unaffected.
- **Revisit trigger**: if an early-iteration parity-net pass reveals the dispatch paths have already diverged in ways that make convergence cheaper to do *first* (i.e., the reorg/DRY work is fighting the fork), re-run `/pipeline-strategy` to reconsider Path A vs B ordering.

## Convergence sequencing — refined by Stage-4 scoping (2026-06-23)

A read-only Stage-4 investigation (the "first-principles grep before architecting" canon) scoped the convergence before any code and **partially tripped the revisit trigger** — direction is sound (proceed iteration-by-iteration; do NOT re-strategize) but the original "WS-mount first" sequencing was wrong. Findings:

- **`ViewRuntime.dispatch_mount` (`runtime.py:303`) / `dispatch_event` (`runtime.py:512`) already exist but have ZERO production call sites** — only `dispatch_message` is wired (WS `url_change` `websocket.py:4824`; SSE `/message/` `sse.py:1192`). They are an unexercised **fourth** parallel copy that has **already drifted** from the WS handlers.
- **The runtime path is missing controls the WS path has** — `dispatch_mount` lacks the post-mount `check_object_permission` (a latent IDOR-class gap, not currently live because nothing mounts through it), and the runtime drains **3 of WS's 8 flush queues** (`runtime.py:1038` vs `websocket.py:888` `_flush_all_pending`) — a live #1646 instance *inside* the supposed cure.
- **SSE doesn't use the runtime to mount** — the SSE stream GET still mounts via legacy `_sse_mount_view` (`sse.py:996`) and `/event/` via `_sse_handle_event` (`sse.py:1120`). So the runtime is unproven against any real mount/event transport.
- **A thin-shim of WS-mount is INFEASIBLE** (confirmed #1853): the mount delta is ~20 behaviors, ~8 of them transport-agnostic state the runtime lacks. Convergence must be `run_pre_mount_auth`-style **sequence extraction**, never a thin shim.
- **The parity nets pin security re-forks, not behavioral parity** — they must grow flush-queue-count / object-perm / version-stamping nets BEFORE the migration, or convergence silently inherits the runtime's gaps.

**Refined split-foundation order (supersedes the "WS-mount first" implication above):**

- **Iter 0 — behavioral-parity nets + runtime gap-fixes** (Low risk / S). Add flush-queue-count, object-permission, and wire-version parity nets; fix the runtime's gaps (add the 5 missing flush queues; add `check_object_permission` to `dispatch_mount`); add a count-test enumerating the WS-only behaviors. Each a standalone quality win; closes the latent object-perm gap before anything routes through the runtime.
- **Iter 1 — converge SSE legacy mount/event onto the runtime** (Low-Med / M). SSE has ~no WS-only behaviors, so this proves the runtime against a real transport and forces it to grow the transport-agnostic hooks WS needs later — a soak release before the high-blast WS work. No WS changes.
- **Iter 2 — WS event convergence** (High / L) — sequence-extraction (not thin shim); real `WebsocketCommunicator` tests; add `event` to `RUNTIME_OWNED_VERBS` last, gate-off-verified.
- **Iter 3 — WS mount convergence** (High / L) — same shape; WS-only mechanics (Channels groups, presence, ticks, actors, sticky `sticky_hold`, `mount_batch`) stay as hooks; module-org refactor of `websocket.py` rides here. Add `mount` to `RUNTIME_OWNED_VERBS` last.

Per-iteration soak (#1122) between each. Iter 0 is the executable first PR for v1.1.0-3.
