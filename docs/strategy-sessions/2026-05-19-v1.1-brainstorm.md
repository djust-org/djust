# v1.1 Brainstorm — Portfolio Menu

**Date:** 2026-05-19
**Status:** Brainstorm — not yet a strategy decision. This document enumerates candidates and rates them; it does not yet commit a scope. The follow-up is either (a) a `/pipeline-strategy` deep-mode session over a chosen subset, or (b) a user-driven pick from this menu.

## Inputs

- **v1.0.0** is shipping (rc6 cut 2026-05-19; rc1–rc6 soak complete; zero open issues at GA).
- **`v1.1.0` ROADMAP slot is currently empty** post-rc6 — the one parked issue (#1434 native async ORM) was audited in rc5 and closed as `not planned`.
- **Already-committed post-1.0 work** (from the 2026-05-17 strategy session + 5 deferred ADRs):
  - **Dead View / Progressive Enhancement** (graceful no-JS fallback)
  - **AI / server-driven arc** — ADR-002 phases 4–5 (broadcast_commands, AssistantMixin), ADR-003 (LLM provider abstraction), ADR-004 (Undo for LLM-driven actions), ADR-005 (Consent envelope for remote control), ADR-006 (AI-generated UIs with capture-and-promote)
- **Workspace context** — djust.org (built on djust, dogfood), djustlive (PaaS), djust-monitor (observability), maxcompanion (mobile POC), djust-orchestrator (automation).
- **Manifesto principles** that should weight ratings — "Complexity Is the Enemy" (penalize XL ideas without a clear win), "Developer First", "AI-Ready by Design" (boost AI ideas), "One Stack, One Truth" (penalize parallel-stack ideas), "Performance Is Architecture", "Own Your Stack", "Opinionated Where It Matters".

## Rating dimensions

| | What it measures |
|---|---|
| **Effort** | S = days · M = 1–2 weeks · L = 3–6 weeks · XL = multi-month |
| **Novelness** | How differentiated vs existing Python web frameworks. Low = me-too · Med = improvement · High = genuinely new |
| **Usefulness** | How many users it helps. Low = niche · Med = many users · High = everyone |
| **Strategic** | Does it advance djust's position? Low = neutral · Med = supports thesis · High = changes what djust is |
| **Risk** | Could it backfire? Low = additive · Med = architectural · High = could fork the project |

Each idea gets a **composite score**: `(Novelness + Usefulness + Strategic) - Effort - Risk`, on a rough scale where Low=1, Med=2, High=3, S=1, M=2, L=3, XL=4. Tie-broken by Strategic value.

## Cluster paths — four distinct 1.1 directions

### Path 1 — **AI-Ready 1.1** (manifesto-load-bearing)

Ship the slice of the AI arc that makes djust the obvious choice for AI-driven apps in 2026.

- **F3** — AssistantMixin minimal (ADR-002 ph5)
- **B6** — LLM token streaming UX
- **F4** — LLM provider abstraction (lightweight, single-provider OK at first)
- **F2** — AI-assisted a11y review (showcase the AssistantMixin)
- **D7** — AI-assisted view debugging (showcase #2)

**Total effort:** ~2 months. **Novelness:** Very high. **Risk:** Med (LLM landscape moves). **Bet:** djust 1.1 is the first Python web framework with first-class AI primitives; the 2026 launch story writes itself.

### Path 2 — **Developer Experience 1.1** (loved by power users)

Ship the quality-of-life features the 1.0 launch will surface demand for.

- **D1** — `djust doctor` v2 interactive dashboard
- **D2** — Template-handler type-check
- **D4** — `djust new` scaffolder
- **D8** — `djust audit` GitHub Action
- **D10** — Inline benchmarks in dev mode
- **D5** — VS Code extension (stretch)
- **H5** — Docs versioning

**Total effort:** ~6–8 weeks. **Novelness:** Low–Med. **Risk:** Low. **Bet:** djust 1.1 is the framework that's most pleasant to use; adoption inflects on DX, not features.

### Path 3 — **Platform / Production 1.1** (enterprise + paid path)

Make djust the obvious choice for teams shipping it in production.

- **A6** — djust-monitor productionization (observability)
- **A7** — djustlive PaaS first-class deploy story
- **B2** — First-class background-job UX
- **G1** — Feature flags / A/B testing
- **G4** — Production health endpoints
- **E2** — OpenTelemetry + Sentry adapters

**Total effort:** ~6–8 weeks. **Novelness:** Med. **Risk:** Low–Med (conflates OSS/SaaS lines). **Bet:** djust 1.1 is the framework production teams adopt; the $ story strengthens.

### Path 4 — **Debug & Time-Travel 1.1** (small, differentiated)

Bet on debugging UX as the killer feature; ship a small, sharp 1.1.

- **B7** — Time-travel sharable URLs ("Replay.io for djust")
- **D1** — `djust doctor` v2 interactive
- **D7** — AI-assisted view debugging
- **D10** — Inline benchmarks in dev mode

**Total effort:** ~4–5 weeks. **Novelness:** High. **Risk:** Low. **Bet:** djust 1.1 ships the most powerful web-app debugger in the Python ecosystem; the "I had to switch to djust because debugging the React app was killing me" demo writes itself.

## Contrarian picks worth surfacing

These ranked lower than the top 10 but are worth a second look because they're high-novelness or open new positioning:

1. **B3 — Replace `django.contrib.admin`** (score 2). Audacious. A djust-driven CRUD admin would be the highest-leverage way to onboard every existing Django user — they all use admin. Risk is scope (admin is a deep surface). Could be a multi-release arc.
2. **B7 — Time-travel sharable URLs** (top 10 already; flagging again — this is THE killer demo idea). Imagine the launch post: "we filed a bug, sent a time-travel URL, the maintainer reproduced it in 0 minutes."
3. **F2 — AI-assisted a11y review** (top 10; flagging — pairs naturally with the framework-wide a11y story shipped in 1.0).
4. **D7 — AI-assisted view debugging** (top 10; flagging — concrete leverage of LLMs that doesn't require committing to the full AI arc).
5. **B6 — LLM token streaming UX** (top 10; flagging — every AI app needs this; djust shipping a clean primitive becomes load-bearing).
6. **A4 — djust-mobile** (score 1; under-rated). The maxcompanion POC exists. A "djust 1.1 ships native mobile bindings" headline is bigger than any DX improvement. Better as a parallel track (a separate `djust-native` project) than crammed into the 1.1 milestone — but worth deciding deliberately.

## What's missing from this brainstorm

Honest acknowledgments — things this menu does NOT cover:

- **Funding / sustainability strategy.** No "make money from djust" line item except A7 (djustlive). The OSS sustainability question is real and not addressed.
- **Community / governance.** No "open up the project to outside contributors" line item. The 5-month-quiet-dev posture has limits.
- **Branding / marketing beyond 1.0.** The 1.0 launch is in the launch package; 1.1's marketing story is downstream of which path you pick.
- **Specific competitor responses.** Phoenix LiveView 1.x evolution, React Server Components maturity, HTMX 2.x — none are explicitly tracked here.

## Recommendation

This brainstorm produces three serious candidate paths (1, 2, 3 above) and one niche-but-sharp option (path 4). The bottom 10 are explicitly NOT recommended.

**My recommendation, with stated bias:**

**Path 1 (AI-Ready 1.1) is the highest-EV move** for the following reasons:

1. The manifesto says "AI-Ready by Design." 1.0 doesn't ship the AI arc. 1.1 is the natural moment.
2. The AI landscape in 2026 will not slow down. Every Python framework will eventually have to take a position on AI primitives. Being first matters; being last is fatal.
3. The slice (F3 + B6 + F4 + F2) is achievable in ~2 months — not the full ADR-002 → ADR-006 multi-quarter arc, just the showcase MVP that makes the story real.
4. It composes with Path 2/3/4 in subsequent milestones — Path 1 doesn't preclude later DX or platform work.

**But Path 2 (Developer Experience 1.1) is the safer recommendation** if 1.0 launch reveals adoption friction. If r/django and r/python feedback says "the docs are great but my IDE doesn't autocomplete `dj-click`" — Path 2 is the right response.

**Decision-shaped next step:** schedule a `/pipeline-strategy` deep-mode session over Path 1 vs Path 2 vs Path 3 with one week of soak time after the 1.0 launch (so launch feedback can weight the choice). The strategy session would also:
- Verify scope by tasking each unit through `/pipeline-next` dry-runs.
- Flag any ADR that needs an update or a new ADR (Path 1 in particular probably needs an ADR-007 for the AssistantMixin shape).
- Convert the chosen path into a `ROADMAP.md v1.1.0` milestone with named work units and a sequencing decision.

## Footnotes

- Composite scores are rough heuristics; treat them as a starting point, not a verdict.
- Several ideas (B11, B12, D3, E1, E6, H3) are quick wins that could ride alongside any chosen path as "fillers" between larger work units.
- The bottom-10 list is deliberately opinionated — they're not bad ideas in the abstract; they're bad fits for djust 1.1 given current scope and manifesto alignment.
- This brainstorm is bigger than 1.1 can carry. Whichever path is chosen, 80% of these ideas remain for 1.2 / 1.3 / 1.x.
