## Summary

This PR implements the `@client_state` decorator as part of Phase 5 of the djust state management system. It adds client-side state coordination for multi-component LiveView applications using a pub/sub pattern (StateBus), along with the `@cache` decorator for result caching with TTL.

## What This PR Does

### Primary Features

1. **StateBus (Pub/Sub Pattern)**: A centralized state management system for coordinating state across multiple components
   - Key-value storage with get/set operations
   - Subscribe/notify pattern for reactive updates
   - Error-resilient subscriber callbacks (errors in one don't affect others)
   - Unsubscribe mechanism via cleanup function
   - Debug logging for all operations

2. **@client_state Decorator**: Python decorator that integrates with StateBus
   - Automatically updates StateBus when event handlers execute
   - Extracts `value` or `checked` from event data
   - Notifies all subscribers of state changes
   - Still calls server for state persistence (eventual consistency)

3. **@cache Decorator**: Client-side result caching with TTL
   - Reduces server load by caching repeated queries
   - Configurable TTL (default 60s)
   - Cache key generation from handler name + parameters
   - Automatic expiration and cleanup
   - Manual cache invalidation support

### Implementation Details

**JavaScript Client** (`decorators.js`):
- `StateBus` class with Map-based storage
- `clientStateEvent()` function for decorator integration
- `cacheEvent()` function for caching logic
- `generateCacheKey()` for cache key generation

- Global singleton `globalStateBus` instance

**Embedded JavaScript** (`live_view.py`):
- Synchronized duplicate implementation for runtime
- StateBus class embedded in LiveView HTML
- All cache and client_state functions duplicated

### Issues & Recommendations

#### Critical Issues ❌

#### Minor Issues ⚠️

1. **Missing client_state.test.js File**
   - **Issue**: PR description claims "31 new tests in tests/js/client_state.test.js" and shows 155 tests passing, but the file doesn't exist in the repository
   - **Impact**: Cannot verify client_state test claims

   - **Resolution**: The file exists in the PR branch (`git show` confirms), but may need to be checked out or merged
   - **Severity**: Minor - file exists in PR diff, just needs to be committed/pushed

2. **Potential Memory Growth in StateBus**
   - **Issue**: No built-in limit on number of state keys or subscribers
   - **Scenario**: Long-lived SPA could accumulate state keys
   - **Mitigation**: `clear()` method exists but no automatic cleanup
   - **Recommendation**: Consider LRU eviction or max key limit (future enhancement)
   - **Severity**: Low - unlikely in typical usage

3. **Cache Key Collision Potential**
   - **Issue**: Cache keys use colon separator which could collide with param values containing colons
   - **Example**: `search:user:123` vs `search:user:1:23` could be ambiguous
   - **Current Code**: `return \`${eventName}:${paramValues}\`;`
   - **Recommendation**: Use URL-safe encoding or JSON stringification for params
   - **Severity**: Low - edge case, unlikely in practice

4. **No Cache Size Limit**
   - **Issue**: `resultCache` has no maximum size constraint
   - **Impact**: Could grow unbounded in long-running sessions
   - **Mitigation**: TTL-based expiration + `cleanupExpiredCache()`
   - **Recommendation**: Add LRU eviction policy (future enhancement)
   - **Severity**: Low - TTL prevents indefinite growth

**Example** (Cache key generation):
```javascript
function generateCacheKey(eventName, eventData, keyParams) {
    if (keyParams.length === 0) {
        return eventName;
    }

    const paramValues = keyParams.map(param => {
        const value = eventData[param];
        if (value === undefined || value === null) {
            return '';
        }
        return String(value);
    }).join(':');

    return `${eventName}:${paramValues}`;
}
```

## Performance Impact

### Expected Performance Gains

1. **@cache Decorator**:
   - **Benefit**: Eliminates redundant server calls for repeated queries
   - **Expected**: 80%+ reduction in server calls for autocomplete/search
   - **Mechanism**: O(1) Map lookup vs network round-trip
   - **Tradeoff**: Client memory usage (mitigated by TTL)

2. **@client_state Decorator**:
   - **Benefit**: Sub-millisecond cross-component updates
   - **Expected**: <10ms state propagation (per IMPLEMENTATION_PHASE5.md)
   - **Mechanism**: In-memory pub/sub vs server round-trip
   - **Tradeoff**: None significant

### Potential Performance Concerns

1. **Memory Usage**:
   - resultCache grows with unique cache keys
   - StateBus grows with state keys and subscribers
   - **Mitigation**: TTL-based expiration, clear() on unmount

2. **Subscriber Notification Overhead**:
   - O(n) where n = number of subscribers per key
   - Could be slow if many subscribers
   - **Mitigation**: Try/catch prevents cascade failures

3. **JavaScript Bundle Size**:
   - +289 lines in decorators.js
   - +197 lines embedded in live_view.py
   - **Current**: ~7.1 KB client.js (per Phase 5 docs)
   - **Target**: Keep under 10 KB

### Performance Testing Recommendations

1. Benchmark cache hit rate in real application
2. Measure StateBus notification latency with 10+ subscribers
3. Monitor memory growth in long-lived sessions
4. Profile JavaScript bundle size after minification
