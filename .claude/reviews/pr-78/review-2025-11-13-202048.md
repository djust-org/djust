## Summary

This PR adds comprehensive health check functionality to the state backend system, enabling production monitoring, Kubernetes readiness/liveness probes, and load balancer health checks. The implementation adds an abstract `health_check()` method to the `StateBackend` base class with concrete implementations for both `InMemoryStateBackend` and `RedisStateBackend`.

**Impact**: Medium

### Core Changes

1. **Abstract Method in StateBackend**
   - Adds `health_check()` abstract method to `StateBackend` base class
   - Defines standard health check response format with status, latency, and backend-specific details

2. **InMemoryStateBackend Implementation**
   - Performs basic cache operations test (read/write/delete)

   - Comprehensive error handling for connection failures

4. **Comprehensive Test Coverage**
   - 5 new tests covering healthy/unhealthy states, latency measurement, and error conditions
   - Tests for both backend implementations
   - Validates response format and data correctness

### Issues & Recommendations

#### Critical Issues ❌
NONE - Code is production-ready as-is.

#### Minor Issues ⚠️

1. **Test Key Cleanup in InMemoryStateBackend**
   ```python
   # Current implementation:
   del self._cache[test_key]
   ```
   - If the delete operation fails, the test key remains in cache
   - **Recommendation**: Wrap cleanup in try/except or use try/finally
   ```python
   try:
       # Test write
       self._cache[test_key] = (None, time.time())
       # Test read
       _ = self._cache.get(test_key)
   finally:
       # Ensure cleanup even if operations fail
       self._cache.pop(test_key, None)
   ```
   - **Impact**: Very low - test key is small and gets cleaned up eventually

2. **Redis Test Key TTL**
   ```python
   self._client.setex(test_key, 10, b"health_check")
   ```
   - 10-second TTL for health check test key seems arbitrary
   - **Recommendation**: Consider shorter TTL (1-2 seconds) since key is deleted immediately
   - **Impact**: Minimal - key is explicitly deleted anyway

3. **Incomplete Test Coverage for InMemory Failure Case**
   - Tests don't verify unhealthy status when InMemory operations fail
   - **Recommendation**: Add test that mocks cache failure to verify error path
   - **Impact**: Low - error handling is implemented, just not tested

#### Nice-to-Have Improvements 💡

1. **Add Example View/Endpoint**
   - Could add a Django view example in documentation showing how to expose health check as HTTP endpoint

## Performance Impact

**Impact**: Negligible to Positive

### In-Memory Backend
- **Overhead**: Sub-millisecond (simple dict operations)
- **Memory**: Minimal (one temporary test key)
- **Impact**: No production performance impact

### Redis Backend
- **Overhead**: 1-5ms typical (PING + 3 operations + INFO)
- **Network**: 1 PING + 1 SETEX + 1 GET + 1 DELETE + 2 INFO = 6 round trips
- **Optimization Opportunity**: Could pipeline operations to reduce round trips
- **Impact**: Acceptable for health check use case (not called frequently)

### Recommendations
- Health checks typically run every 5-30 seconds
- Current implementation is well-suited for this frequency
- 💡 If sub-millisecond health checks are needed, could add a cached/debounced option
