## What This PR Does

### Python API (`python/djust/drafts.py`)

Introduces `DraftModeMixin` class that can be mixed into any LiveView:

```python
class ArticleEditorView(DraftModeMixin, LiveView):
    template_name = 'editor.html'
    draft_enabled = True  # Enable draft mode (default: True)
    draft_key = 'article_editor'  # Custom draft key (optional)

    def save_article(self, title, content):
        Article.objects.create(title=title, content=content)
        self.clear_draft()  # Clear draft on successful save
```

**API Surface:**
- `draft_enabled: bool = True` - Enable/disable draft mode
- `draft_key: Optional[str] = None` - Custom localStorage key
- `get_draft_key() -> str` - Get draft key (can be overridden for dynamic keys)
- `clear_draft()` - Clear draft from localStorage
- `get_context_data()` - Adds draft config to template context

### JavaScript Client (`decorators.js` + `live_view.py`)

**DraftManager Class:**
- `saveDraft(key, data)` - Debounced save (500ms)
- `loadDraft(key)` - Load from localStorage
- `clearDraft(key)` - Remove draft
- `getAllDraftKeys()` - List all draft keys
- `clearAllDrafts()` - Remove all drafts

**Helper Functions:**
- `collectFormData(container)` - Extract form data from container
- `restoreFormData(container, data)` - Restore form data to container
- `initDraftMode()` - Initialize draft tracking on page load

**Supported Elements:**
- `<input>` (text, checkbox, radio, etc.)
- `<textarea>`
- `<select>`
- `<div contenteditable="true">` (rich text editors)

### Template Integration

Templates need minimal markup to enable draft mode:

```html
<div data-liveview-root
     data-draft-enabled="{{ draft_enabled }}"
     data-draft-key="{{ draft_key }}">

    <input name="title" data-draft="true" />
    <textarea name="content" data-draft="true"></textarea>
</div>
```

### Issues & Recommendations

#### Critical Issues ❌

#### Minor Issues ⚠️

1. **No User Confirmation for Draft Restoration**
   - Currently auto-restores drafts without asking user
   - Some apps may want to show "Resume draft from X minutes ago?" prompt
   - **Recommendation**: Consider adding optional `draft_prompt: bool` config in future
   - **Not blocking**: Current behavior (auto-restore) is reasonable default

2. **No Server-Side Draft Storage Option**
   - Only localStorage (client-side) supported
   - Multi-device scenarios won't sync drafts
   - **Recommendation**: Document this limitation in `STATE_MANAGEMENT_API.md`
   - **Not blocking**: Marked as "optional - not implemented" in IMPLEMENTATION_PHASE5.md

3. **Limited Example Coverage in Demo Project**
   - Only one example view (`draft_mode_test.py`)
   - Would benefit from rich text editor example
   - **Recommendation**: Add `rich_editor_example.py` showing contenteditable usage
   - **Not blocking**: Test page demonstrates all features adequately

#### Nice-to-Have Improvements 💡

1. **Draft Age Display**
   - Could show "Draft saved 2 minutes ago" in UI
   - Timestamp is stored (`draftData.timestamp`) but not exposed to template
   - **Future enhancement**: Add `draft_age` to context for display

2. **Draft Size Warning**
   - localStorage has 5-10MB limit per domain
   - Large drafts (e.g., pasting huge text) could fail silently
   - **Future enhancement**: Add `max_draft_size` config with warning

3. **Namespace Collision Protection**
   - All drafts share `djust_draft_` prefix
   - Multiple views with same `draft_key` would collide
   - **Current mitigation**: Developer sets unique `draft_key` per view
   - **Future enhancement**: Include view class name in auto-generated key

def get_draft_key(self) -> str:
    """
    Get the draft key for localStorage.
    Override this method to customize the draft key based on view state.

    if self.draft_key:

    # Default: Use view class name

## Security Considerations

### localStorage Security

**✅ No Security Issues:**
- localStorage is origin-scoped (same-origin policy)
- Data not transmitted to server (no network exposure)
- No XSS risk (data sanitized on restore)
- No CSRF risk (purely client-side)

**Considerations:**
- Drafts stored in plaintext in browser
- Shared computer users could see each other's drafts
- Browser security bugs could expose localStorage
- **Recommendation**: Document in security guidelines that sensitive data (passwords, SSNs) should NOT use draft mode

### Input Sanitization

**✅ Proper Handling:**
- Field values are assigned via `.value` or `.checked` (safe)
- contenteditable uses `.innerHTML` (potential XSS if draft contains malicious HTML)
- **Mitigation**: Server-side validation on submit handles this
- **Not a regression**: No worse than user typing directly

### Data Persistence

**✅ No Privacy Issues:**
- Drafts stored locally (user's device)
- No server logging of draft content
- Can be cleared via `clear_draft()` or browser settings
- **Recommendation**: Add "Clear All Drafts" button in demo project
