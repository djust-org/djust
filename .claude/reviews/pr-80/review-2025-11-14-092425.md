## Summary

This PR implements the `@cache` decorator for Phase 5 of djust's state management system, adding client-side result caching with TTL-based expiration. The implementation provides a clean, Pythonic API for caching expensive event handler results on the client, reducing server load and improving perceived performance. The decorator integrates seamlessly with the existing state management decorators (@debounce, @throttle, @optimistic) and follows the established pattern of keeping logic in JavaScript while exposing a simple Python decorator API.

The implementation is production-ready with comprehensive test coverage (27 new tests, 124 total passing), proper dual implementation maintenance (ES module + embedded JavaScript), and clear documentation. The cache system uses intelligent key generation from handler names and parameters, respects configurable TTL values, and provides manual cache invalidation capabilities.

**Impact**: Medium
**Changes**: 1,347 additions, 1 deletion across 4 files

## What This PR Does

### Core Functionality

1. **Client-Side Caching**: Implements a `resultCache` Map that stores event handler results keyed by handler name and parameters
2. **TTL-Based Expiration**: Configurable time-to-live (default 60s) with automatic expiration checking on cache lookups
3. **Parameterized Cache Keys**: Generates unique cache keys from handler name plus specified parameters (e.g., `search:query1:page2`)
4. **Cache Hit/Miss Logic**: Checks cache before making server calls, stores results on cache miss
5. **Manual Invalidation**: Provides `clearCache()` for selective or full cache clearing
6. **Automatic Cleanup**: Includes `cleanupExpiredCache()` for removing expired entries
7. **Debug Logging**: Comprehensive logging showing cache hit/miss/cleanup with remaining TTL

### Usage Example

```python
from djust import LiveView
from djust.decorators import cache, debounce

class AutocompleteView(LiveView):
    template_name = "autocomplete.html"

    def mount(self, request):
        self.query = ""
        self.results = []

    @debounce(wait=0.3)
    @cache(ttl=300, key_params=["query"])
    def search(self, query: str = "", **kwargs):
        """Cached search with 5 min TTL"""
        self.query = query
        if query:
            self.results = Product.objects.filter(
                name__icontains=query
            )[:10]
        else:
            self.results = []
```

### Issues & Recommendations

#### Critical Issues ❌

**NONE** - No blocking issues identified. The implementation is production-ready.

#### Minor Issues ⚠️

1. **No LRU Eviction Policy**
   - **Issue**: Cache can grow unbounded if many unique queries are made
   - **Impact**: Low - TTL expiration provides eventual cleanup, but high-traffic apps could accumulate entries
   - **Recommendation**: Consider adding max cache size with LRU eviction (e.g., 100 entries max)
   ```javascript
   const MAX_CACHE_SIZE = 100;
   if (resultCache.size >= MAX_CACHE_SIZE) {
       // Evict oldest entry
       const firstKey = resultCache.keys().next().value;
       resultCache.delete(firstKey);
   }
   ```

2. **No Periodic Cleanup**
   - **Issue**: `cleanupExpiredCache()` is defined but not automatically called
   - **Impact**: Low - Expired entries only cleaned on manual call or when accessed
   - **Recommendation**: Add periodic cleanup (e.g., every 5 minutes)
   ```javascript
   // In LiveViewWebSocket constructor
   setInterval(() => cleanupExpiredCache(), 5 * 60 * 1000);
   ```

3. **Cache Key Collision Risk**
   - **Issue**: Different handlers with same name could collide if not namespaced
   - **Impact**: Very Low - Unlikely in practice, but theoretically possible
   - **Recommendation**: Consider including component/view ID in cache key for multi-component apps

#### Nice-to-Have Improvements 💡

1. **Cache Statistics**: Add metrics for monitoring cache performance
   ```javascript
   export const cacheStats = {
       hits: 0,
       misses: 0,
       evictions: 0,
       size: () => resultCache.size
   };
   ```

2. **Cache Warming**: Support for pre-populating cache on mount
   ```python
   @cache(ttl=300, warm=True)
   def search(self, query: str = ""):
       # Called on mount with default params
   ```

3. **Stale-While-Revalidate**: Return cached result while fetching fresh data
   ```javascript
   if (cached && Date.now() < cached.staleAt) {
       // Return stale data immediately, fetch fresh in background
   }
   ```

4. **Cache Versioning**: Invalidate cache when code changes
   ```javascript
   const CACHE_VERSION = 'v1';
   const cacheKey = `${CACHE_VERSION}:${eventName}:${params}`;
   ```

## Security Considerations

### Security Assessment: ✅ No Issues

**Reviewed Areas**:

1. **Cache Poisoning**: Not a concern
   - Cache is client-side only
   - Each user has their own cache
   - No cross-user cache sharing

2. **Sensitive Data Exposure**: Low risk
   - Cache stored in JavaScript memory (not localStorage)
   - Cleared on page reload
   - No persistence to disk

3. **Cache Key Injection**: Not vulnerable
   - Keys are simple strings
   - No code execution from keys
   - No special character handling needed

4. **DoS via Cache Fill**: Mitigated
   - TTL provides automatic cleanup
   - Recommendation: Add LRU eviction for extra protection

**Best Practices**:
- Don't cache sensitive data (passwords, tokens, etc.)
- Use short TTL for user-specific data
- Clear cache on logout (if implemented)

## Decorator Stacking Behavior

### Analysis: Works Correctly with Other Decorators

**Common Combinations**:

1. **@debounce + @cache**: ✅ Correct order
   ```python
   @debounce(wait=0.3)  # Applied first (outer)
   @cache(ttl=300)      # Applied second (inner)
   def search(self, query: str):
       pass
   ```
   - Debounce delays the call
   - Then cache checks before server call
   - Optimal for autocomplete/search

2. **@throttle + @cache**: ✅ Works correctly
   ```python
   @throttle(interval=1.0)
   @cache(ttl=60)
   def refresh_data(self):
       pass
   ```
   - Throttle limits frequency
   - Cache provides additional deduplication

3. **@optimistic + @cache**: ⚠️ Use carefully
   ```python
   @optimistic
   @cache(ttl=300)
   def submit_form(self, data):
       pass
   ```
   - Optimistic update happens first
   - Cache may show stale optimistic state
   - Recommendation: Don't cache @optimistic handlers

**Recommendation**: Add section to STATE_MANAGEMENT_PATTERNS.md documenting decorator stacking patterns.
