## Summary

This PR fixes a `TypeError` that occurred when adding component instances (specifically `NavbarComponent`) to template context. The issue was that component classes lacked `__str__()` methods, preventing them from being properly JSON serialized by `DjangoJSONEncoder`. The PR adds `__str__()` methods to all component base classes and updates the JSON encoder to handle components from both old and new locations.

The fix is well-targeted and addresses the immediate issue, but the PR has **one critical test failure** that must be resolved before merging. Additionally, the PR introduces significant changes to the template inheritance extraction logic that appear to be unrelated to the stated fix and may be causing the test failure.

**Type**: Bugfix
**Impact**: Medium
**Changes**: 498 additions, 46 deletions across 6 files

---

## What This PR Does

### Core Fix
1. **Added `__str__()` methods** to three component classes:
   - `Component` (old location: `python/djust/component.py`)
   - `LiveComponent` (old location: `python/djust/component.py`)
   - `LiveComponent` (new location: `python/djust/components/base.py`)

2. **Updated `DjangoJSONEncoder`** to handle components from both old and new import paths

3. **Fixed template usage** in the demo project to use `{{ navbar }}` instead of `{{ navbar.render }}`

### Additional Changes
4. **Modified template inheritance extraction logic** in `LiveView.get_template()` - this appears unrelated to the component serialization bug
5. **Added `_strip_liveview_root_in_html()` method** - unclear why this is needed for the stated bug fix
6. **Updated regex patterns** for `data-liveview-root` extraction to allow attributes in any order
7. **Created comprehensive test suite** with 13 tests (but tests aren't in the repository yet based on the diff location)

### Critical Issues ❌

1. **Test failure blocking merge**
   ```
   FAILED tests/unit/test_live_view.py::TestTemplateInheritance::test_get_template_with_extends
   AssertionError: assert '<!DOCTYPE html>' not in '<!DOCTYPE html><html>...'
   ```
   - The test expects `get_template()` to NOT return the full HTML with DOCTYPE
   - But changes to template extraction logic in `live_view.py` (lines 238-253) now return the full merged template
   - This appears to be a regression caused by unrelated refactoring in this PR

2. **Scope creep** - PR includes substantial refactoring of template inheritance logic that is **not related to the component serialization bug**:
   - Replaced block extraction logic with merged template extraction (~40 lines changed)
   - Added new `_strip_liveview_root_in_html()` method (60+ lines)
   - Modified VDOM template processing flow
   - These changes should be in a separate PR focused on template inheritance

3. **Test file location issue** - The diff shows `python/djust/tests/test_component_rendering.py` as a new file (392 lines), but this doesn't follow the project's test directory structure. Existing tests are in `tests/unit/`, not `python/djust/tests/`

### Minor Issues ⚠️

1. **Inconsistent JSON encoder imports** (line 44-47 in `live_view.py`):
   ```python
   from .component import Component as OldComponent, LiveComponent as OldLiveComponent
   from .components.base import Component, LiveComponent
   ```
   - Importing same class names with aliases is confusing
   - Consider a cleaner pattern, e.g., checking `isinstance` with a tuple or using `hasattr(obj, 'render')`

2. **Missing type hints** - The new `__str__()` methods don't have return type annotations (should be `-> str`)

3. **Whitespace-only changes** - Several files have trailing whitespace removals mixed with functional changes (e.g., `views_old.py` lines 559, 971, 993, 998, 1098, 1111). These should ideally be in a separate cleanup commit.

4. **Template string update incomplete** - The fix updates the demo template to use `{{ navbar }}`, but the view code change (line 106 in `views_old.py`) explicitly calls `.render()` before adding to context:
   ```python
   context['navbar'] = navbar.render()
   ```
   This means the `__str__()` method is never actually called in the demo. Either:
   - Use `context['navbar'] = navbar` (let `__str__()` do the work), or
   - Keep calling `.render()` explicitly (but then `__str__()` won't be exercised)

### Nice-to-Have Improvements 💡

1. **Consider deprecation path** - If there are both old and new component locations, consider adding deprecation warnings to guide users toward the new location

2. **Add integration test** - While the new tests are comprehensive, they don't test the actual bug scenario: adding a component to context and JSON encoding it in a real view

3. **Performance consideration** - `__str__()` now calls `render()` every time the object is converted to string. For components used multiple times (e.g., in loops), consider caching the rendered HTML or documenting the performance implications

4. **Documentation update** - Update `COMPONENT_EXAMPLES.md` or similar docs to mention that components can now be used directly in templates without calling `.render()`

## Performance Impact

### Minimal Performance Impact

1. **`__str__()` overhead**: Negligible - only called when object is converted to string
2. **JSON encoding**: No change - still renders to HTML string, just via different code path
3. **Template rendering**: Slightly more efficient since we can use `{{ component }}` instead of `{{ component.render }}`

### Potential Concern

The template inheritance refactoring changes how templates are processed:
- Old: Extract child template blocks only
- New: Use full merged template, then extract liveview-root, then strip comments/whitespace

This could have **performance implications** for complex templates with deep inheritance chains. Needs benchmarking if this refactoring is kept.
