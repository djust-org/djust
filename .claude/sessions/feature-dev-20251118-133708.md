## Phase 2: Enhanced @event_handler Decorator

### Goals
- Auto-extract parameter names, types, and descriptions from handler signatures
- Store metadata in `_djust_decorators` for validation and debug panel
- Maintain backward compatibility with existing code
- Support optional explicit parameter declaration

### Implementation Approach

**Decision**: Enhanced the existing `@event_handler` decorator to support both simple usage (`@event_handler`) and parameterized usage (`@event_handler(description="...")`), with automatic signature introspection.

**Rationale**:
- Backward compatible - existing `@event_handler` usage continues to work
- DRY principle - metadata extracted once from signature, used by validation and debug panel
- Developer experience - auto-documentation via type hints
- Foundation for Phase 3 - metadata ready for debug panel consumption

### Implementation Details

**Decorator Pattern**:
```python
def event_handler(params=None, description=""):
    def decorator(func):
        sig_info = get_handler_signature_info(func)
        # ... extract and store metadata
        return func

    # Support both @event_handler and @event_handler()
    if callable(params):
        func = params
        params = None
        return decorator(func)

    return decorator
```

**Metadata Structure**:
```python
func._djust_decorators = {

        "param_names": ["value", "count"],
        "description": "Search items",
        "accepts_kwargs": True,
        "required": [],
        "optional": ["value", "count"]
    }
}
```

### Key Decisions

1. **Dual decorator syntax support**
   - `@event_handler` - Simple, no parentheses (backward compatible)
   - `@event_handler()` - With parentheses (for explicit params/description)
   - Both use the same code path via clever parameter detection

2. **Metadata reuse**
   - Uses `get_handler_signature_info()` from Phase 1 validation module
   - Same metadata structure used by validation and debug panel
   - Avoids code duplication

3. **Backward compatibility**
   - Existing `_is_event_handler` and `_event_name` markers preserved
   - Existing code continues to work without changes
   - New metadata is additive, not breaking
