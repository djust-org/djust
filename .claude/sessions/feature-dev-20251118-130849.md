## Phase 1: Event Parameter Validation (Core Foundation)

### Goals
- Validate handler signatures before calling
- Provide clear error messages for parameter mismatches
- Support runtime type validation using type hints
- Fail fast on validation errors (strict mode)

### Implementation Approach

**Decision**: Implemented validation as a separate module (`python/djust/validation.py`) that can be imported and used in both WebSocket and HTTP event handlers.

**Rationale**:
- Separation of concerns - validation logic is independent and testable
- Reusable across different transport modes (WebSocket, HTTP)
- Easy to extend for future phases (debug panel can use the same validation utilities)

### Key Decisions

1. **Strict validation by default**
   - Reject unexpected parameters unless handler has **kwargs
   - Fail fast with clear error messages
   - Type validation using runtime type hints

2. **Comprehensive error responses**
   - Include expected vs provided parameters
   - Include type error details
   - Clear, actionable error messages

3. **Foundation for Phase 2**
   - `get_handler_signature_info()` extracts all metadata needed for debug panel
   - Signature includes: params (name, type, required, default), description, accepts_kwargs
