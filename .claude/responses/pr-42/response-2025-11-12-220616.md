### ❓ Question 2: WebSocket Testing

**Review Question:**
> Should we test WebSocket mode before merge, or defer to post-merge testing?

**Answer:**

**Recommendation**: Defer to post-merge testing.

**Rationale:**

1. **Code Path Analysis**:
   - HTTP mode: `handle_event()` → process → return JSON response
   - WebSocket mode: `receive()` → `handle_event()` → process → send JSON message
   - The decorator logic is **identical** in both modes
   - Only the transport layer differs (HTTP request/response vs WebSocket send/receive)

2. **Confidence Level**:
   - High confidence WebSocket mode works correctly
   - All JavaScript code is transport-agnostic
   - Server-side code handles both modes identically
   - VDOM patching and state management are transport-independent

3. **Testing Already Done**:
   - ✅ 97 unit tests (92.81% coverage)
   - ✅ Manual testing in HTTP mode
   - ✅ Automated curl testing
   - ✅ Demo views verified

4. **Risk vs Delay**:
   - Risk: Low (code path identical, high confidence)
   - Delay: Would block merge for verification testing
   - Benefit: Marginal (unlikely to find issues)

5. **Issue Created**:
   - Issue #44 tracks WebSocket testing
   - Can be done post-merge without risk
   - Documented for production readiness checklist

**Decision**: Defer to post-merge. Create Issue #44 for WebSocket testing (already done).

### ❓ Question 3: Dual Implementation

**Review Question:**
> The embedded JS + decorators.js dual implementation is documented. Any concerns about keeping them in sync?

**Answer:**

**Concerns**: Minimal, with proper workflow and documentation.

**Why This Design Works:**

1. **Clear Separation**:
   - **Runtime (embedded JS)**: What users load in browser
   - **Testing (decorators.js)**: What tests import
   - Purpose of each file is crystal clear

2. **Testability Requirement**:
   - Embedded JS cannot be imported by test frameworks
   - ES6 module required for Vitest/Jest imports
   - No other solution enables comprehensive unit testing
   - 97 tests (92.81% coverage) justify this design

3. **Maintenance Workflow**:
   ```
   1. Edit decorators.js (testable version)
   2. Run unit tests → verify correctness
   3. Copy logic to embedded JS
   4. Run manual tests → verify runtime
   5. Both versions now in sync
   ```

4. **Documentation**:
   - Architecture documented in TESTING_JAVASCRIPT.md
   - Implementation tracking in IMPLEMENTATION_PHASE3.md
   - Warning comments in both files
   - Issue #47 will add CONTRIBUTING.md section

5. **Safeguards**:
   - Manual testing catches sync issues quickly
   - Both versions have same functionality
   - Diffs between files are obvious (exports vs no exports)
   - CI runs both unit tests and manual tests

**Mitigation Strategies:**

1. **Short-term** (implemented):
   - Clear comments in both files
   - Architecture documentation
   - Testing guide explains dual implementation

2. **Medium-term** (Issue #47):
   - CONTRIBUTING.md workflow section
   - Checklist for keeping in sync
   - Common pitfalls documented

3. **Long-term** (possible future work):
   - Could explore build-time solution (generate one from the other)
   - Would require tooling complexity
   - Current manual sync is acceptable tradeoff

**Decision**: No major concerns. The dual implementation is well-documented and the testability benefit outweighs the sync overhead. Issue #47 will improve contributor documentation.
